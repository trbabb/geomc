19 /*
20  * random has some design problems:
21  * 1) Templated types (Complex<T>, e.g.) cannot have rand() specializations,
22  * because c++ (for no good reason) does not allow partially-specialized
23  * template functions.
24  * 2) There wouldn't be a way to make a drop-in Random class which, e.g., implements
25  * a low discrepancy sequence, because rand() isn't virtual.
26  * 3) Picking from distributions is mostly unaddressed.
27  * Note that Sampler is, in a way, a tool for picking from distributions over
28  * vectors, though it might also be a sensible thing to pass other
29  * distributions to the sampler for radial density, angular density, etc. etc.
30 
31  * I should probably lookup std::rand and see if maybe that's actually better.
32  * > note that std::rand is c++11 only.
33  *
34  * requirements for new design:
35  - picking from uniform distributions is easy.
36  - adding new types of objects to pick is easy.
37  - it is possible to partially specialize picking of templated objects.
38  - objects can be picked from a source of entropy in a predictable sequence.
39  - low discrepancy sequences have a natural representation.
40  ~ would be nice if swapping out other distributions for uniforms were
41  possible.
42  *
43  * proposal:
44  * rand<type,distribution=UNIT_INTERVAL>() calls RandomHelper<type>::rand()
45  * rand<type>(hi) // half-interval
46  * rand<type>(lo,hi) // full-interval
47  *
48  * (would functors help us?)
49  *
50  * other distributions such as: normal<params> or statified<params> or hammersley<dimensions>
51  * So what you need is:
52  * a way to slip something in between the unit-interval picker and the final result
53  * which:
54  * - doesn't make basic picking slower
55  * - doesn't make uniform interval choices slower
56  * - doesn't make unit and interval choices more difficult or cumbersome to use.
57  *
58  * There is something to be said for simply transforming floats-- assumed to
59  * be uniform-- to shapes with simple functions that are agnostic of the generator.
60  * This is certainly better for stratified/LD sampling.
61  *
62  * The one thing I forbid is constructing a new object specifically to pick from a
63  * uniform distribution. That's too verbose.
64  *
65  * what we have is
66  * entropy -> PRNG -> (random bits) -> uniform real^n -> distribution
67  * kb? MT? statified? uniform?
68  * time? LC? Downey? normal?
69  * network? yarrow? Hammersley? poisson?
70  * device? etc. random? etc.
71  * hard seed?
72  * etc.
73  *
74  * where each of those parts could be interchanged.
75  *
76  * entropy:
77  * PRNG has a method, seed(), which takes a char* and a count, for entropy
78  * input. maybe this is different for each PRNG (e.g. for Yarrow, which
79  * requires an estimate of entropy bits. at the very least,
80  * each PRNG should take a single 32 or 64 bit seed. beyond that, modularity
81  * is not important.
82  * note that Hammersley (?) is completely deterministic and doesn't take a
83  * seed.
84  * PRNG:
85  * this is pretty straightforward, a PRNG's only job is to output random bits.
86  * so having a method to fill an array with bits is maybe a useful thing,
87  * rather than mandating 32 or 64 bits at a time. But providing these methods
88  * shall be provided for bare minimum interoperability. the rand<32> and rand(ct)
89  * methods may be implemented one-in-terms-of-the-other in whichever direction
90  * is convenient for the implementation.
91  * Uniform real:
92  * perhaps this is a template param of rand<T>()? a class to transform random
93  * bits to (0,1)^n.
94  * - may need more than a class name-- instances might be necessary.
95  * e.g. a stratified sampler with parameterized strata counts.
96  * - if not bundled with PRNG, hard to make modular/swappable.
97  * - if bundled with PRNG, stateful.
98  * - Some algorithms are not parameterized (Downey) and some are (stratified).
99  * - some care about how many samples are to be drawn, and some don't.
100  * This makes encapsulation harder.
101  * This is the object that we should pass around and call Random.
102  * Each Random is templated over a single type of object.
103  * Distributions over the reals^N are handled by a specialization of
104  * Random over vectors.
105  * > i.e., we make a Hammersley<T,N> which is an instance of Random<Vec<T,N>>?
106  * - yes, I think this is right. but the last question is the virtual indirection
107  * we'll be adding. can we remove that without adding too much complexity?
108  * Distribution:
109  * Drawing from the uniform distribution should not be hard or require
110  * constructing objects. This should probably be its own class,
111  * and does not need to be "drop-in", in that the client of the random
112  * numbers should be in control of the distribution, and will likely need to
113  * know if it is changing. if modularity is desired, the client should explicitly
114  * provide/expose it.
115  *
116  * basically, code is likely to make the assumption that the chosen numbers are
117  * from a uniform distribution, and we can't allow that to be easily broken by
118  * passing a magically-wrapped Random with some j. random distribution swapped in.
119  *
120  * As such we don't need to implement distributions just yet; they're pretty
121  * orthogonal to the rest of the number generation problem.
122  *
123  * note that really, distributions (gaussian, e.g.) may not pick their numbers
124  * by transforming a uniform real. they might pick based on the pseudorandom
125  * bits directly. This is especially true if we model LD sequences as distributions.
126  *
127  * There is nothing prohibiting us from nesting distributions, e.g. passing a
128  * uniform real distribution to a gaussian distribution. So maybe we should
129  * make 'uniform real' a special type of distribution which is virtual, and that's
130  * what we normally pass around.
131  */
